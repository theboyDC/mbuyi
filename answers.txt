In object-oriented programming, encapsulation is the practice of bundling data with the methods that operate on it while restricting direct external access, typically via private fields and public getters/setters;
you would identify it in code by looking for classes that hide their internal state behind method interfaces, and it is used to protect data integrity and reduce unintended interference.
Abstraction simplifies complex systems by exposing only essential features and hiding implementation details,
which you can recognize through abstract classes or interfaces that define clear contracts without revealing underlying logic,
and it serves to manage complexity and promote code reusability.
Inheritance allows a class to derive properties and behavior from a parent class,
identifiable by the "extends" keyword or subclass relationships,
and it is used to establish hierarchical taxonomies and share common code.
Polymorphism enables objects of different classes to be treated as objects of a common superclass,
most visibly through method overriding or interface implementations where the same method call produces different behaviors, and it is used to write flexible, interchangeable code that can work with diverse object types through a unified interface.

Test-Driven Development (TDD) is a software development practice that follows a short,
repetitive cycle known as "Red-Green-Refactor," where the process begins by writing a failing test for a small,
specific piece of functionality before any production code exists—this initial "Red" stage forces the developer to clearly define the desired behavior and expected outcomes from the user's perspective.
Next, in the "Green" stage, the developer writes the simplest possible implementation code to make that test pass,
focusing solely on satisfying the test conditions without worrying about optimization or elegance.
Finally, in the "Refactor" stage, the developer cleans up the code, removes duplication,
and improves its structure while ensuring all tests continue to pass,
thereby maintaining functionality while enhancing maintainability.
Tests are written before implementation because this practice drives the design from the outside-in,
ensuring that every line of code has a clear purpose and is testable by nature,
which fundamentally influences the quality and design of the code produced—it encourages modular, loosely coupled, and highly cohesive components,
reduces debugging time by catching defects early, and creates a comprehensive safety net of regression tests that gives developers confidence to refactor and extend the system without fear of breaking existing functionality.

The super constructor refers to the constructor of a parent class, and it exists because a subclass inherits the fields and methods of its superclass,
meaning the parent's initialization logic must run before the subclass can add its own properties to ensure the object is properly constructed from the ground up.
In most object-oriented languages, if a subclass constructor does not explicitly call super(),
the compiler automatically inserts a call to the parent's no-argument constructor,
but if the parent lacks such a constructor and only has parameterized constructors,
the subclass will fail with a compilation error because the compiler cannot determine how to initialize the inherited part of the object.
To resolve this, the subclass constructor must explicitly invoke super(parameters) as its first statement, passing the required arguments to match one of the parent's constructors.
For example, if you have a parent class Vehicle with a constructor public Vehicle(String make),
and a subclass Car whose constructor attempts to add a model field, the correct implementation would be public Car(String make,
String model) { super(make); this.model = model; }—without that explicit super(make) call,
the code would fail because Vehicle does not have a default no-argument constructor,
demonstrating that the super constructor is essential for maintaining inheritance integrity and ensuring the parent class is fully initialized before the child class adds its own state.
